## 1. API 요청 흐름 (정상 경로)

```
Client
  │
  ├─ POST /v1/agent/boot ──────────────────────────► handle_boot()
  │         (무인증, DID 발급)                            │
  │                                                    └─ AgentRecord 생성 + WELCOME_CREDITS
  │
  ├─ Authorization: Bearer did:helm:<pubkey>
  │         │
  │         ▼
  │   require_auth() middleware
  │         │── DID 레지스트리에 존재? → 401
  │         │── [PATCH] ed25519 서명 검증 (write ops)
  │         │── [PATCH] Rate limit 체크 (100 req/min)
  │         │── CallerDid 주입 → 핸들러 전달
  │         │
  │         ├─ /v1/sense/cortex ────────────────────► handle_cortex()
  │         │         (QKV-G 어텐션 엔진, G-metric 계산)
  │         │
  │         ├─ /v1/sense/memory/:key ───────────────► handle_memory_*()
  │         │         (DID-격리 KV 스토어, TTL 지원)
  │         │
  │         ├─ /v1/synco/stream ─────────────────────► handle_synco()
  │         │         (GRG 코덱: Golomb-Rice + Redstuff + Golay)
  │         │
  │         ├─ /v1/agent/:did/credit ───────────────► handle_fico()
  │         │         [PATCH] 소유자만 전체 상세 / 타인은 공개 정보만
  │         │
  │         ├─ /v1/agent/:did/earnings ─────────────► handle_earnings()
  │         │         (이미 소유자 체크 존재 ✓)
  │         │
  │         ├─ /v1/pool + /v1/pool/:id/join ─────────► handle_*_pool()
  │         │         [PATCH] 잔고 검증 + monthly_cost 상한
  │         │
  │         └─ /v1/marketplace/post + /apply ────────► handle_*_post()
  │                   [PATCH] 크기 제한 + 중복 지원서 방지
  │
  └─ GET /v1/leaderboard (공개) ──────────────────────► handle_leaderboard()
```

## 2. 인증 레이어 상세

```
요청 도착
    │
    ▼
┌─────────────────────────────────────────────────────┐
│  require_auth() — auth.rs                           │
│                                                     │
│  1. Authorization 헤더 파싱                          │
│     "Bearer did:helm:<base58_pubkey>"               │
│                                                     │
│  2. DID 레지스트리 조회                              │
│     agents.contains_key(&did) → 401 if missing      │
│                                                     │
│  3. [PATCH] ed25519 서명 검증                        │
│     X-Helm-Signature: <base64(ed25519(sha256(body)))│
│     - POST/PUT/DELETE: 서명 헤더 있으면 검증         │
│     - 서명 검증 실패 → 401                          │
│                                                     │
│  4. [PATCH] Rate Limit (슬라이딩 윈도우)             │
│     state.rate_limits: DID → Vec<timestamp_ms>      │
│     60초 윈도우 내 >100 요청 → 429                   │
│                                                     │
│  5. CallerDid extension 주입                        │
└─────────────────────────────────────────────────────┘
```

## 3. 전체 취약점 목록 및 패치 현황

### 🔴 CRITICAL (실제 익스플로잇 가능)

| # | 파일 | 취약점 | 공격 방법 | 패치 |
|---|------|--------|-----------|------|
| C1 | `auth.rs:78` | **DID 서명 검증 완전 부재** | DID 문자열만 알면 타인 위장 가능 | ed25519 X-Helm-Signature 검증 추가 |
| C2 | `boot.rs:122` | **self-referral 가능** | DID-A 생성 후 DID-B를 A로 referral → 15% 수익 사기 | `referrer_did == new_did` 차단 (동일 공개키) |
| C3 | `pool.rs:194` | **무제한 스테이크 (잔고 미검증)** | `stake_virtual: u64::MAX` → bnkr_collected 오버플로 | `agent.virtual_balance >= stake` 검증 + 차감 |
| C4 | `pool.rs:79` | **monthly_cost NaN/Inf 허용** | `monthly_cost: f64::INFINITY` → 수치 오염 | `is_finite()` + 상한 1_000_000 USD |
| C5 | `fico.rs:109` | **타인 FICO 전체 정보 노출** | 임의 DID의 재정 상세(spend, calls) 열람 | 자신만 전체 / 타인은 score+band만 |
| C6 | `auth.rs:90` | **Rate Limit TODO만 있고 미구현** | 1초에 무제한 요청 → DoS | 슬라이딩 윈도우 100 req/60s 구현 |
| C7 | `cortex.rs:204` | **knowledge_context 무제한** | 10GB 컨텍스트 → OOM 크래시 | max 50 items, 각 4096 chars |
| C8 | `synco.rs:124` | **data_b64 크기 무제한** | 1GB base64 → 디코딩 시 메모리 폭탄 | 최대 50MB (base64 문자열 기준) |

## 5. 패치 전/후 공격 시나리오 비교

### 시나리오 A: DID 위장 공격 (C1)
```
[패치 전]
공격자: Authorization: Bearer did:helm:<victim_pubkey>
서버: DID 레지스트리에 있음? YES → 통과
결과: 피해자 계정으로 모든 write 작업 가능

[패치 후]
공격자: Authorization: Bearer did:helm:<victim_pubkey>
서버: X-Helm-Signature 없음 또는 잘못됨 → 401 Unauthorized
결과: 차단됨
```

### 시나리오 B: 무제한 스테이크 잔고 부정 (C3)
```
[패치 전]
공격자: POST /v1/pool/pool-id/join {"stake_virtual": 18446744073709551615}
서버: pool.bnkr_collected += u64::MAX → 오버플로
결과: 즉시 100% 펀딩 달성

[패치 후]
서버: agent.virtual_balance (= 10_000_000 μV) < stake_virtual → 402 Insufficient Balance
결과: 차단됨
```

### 시나리오 C: FICO 파밍 (L3)
```
[패치 전]
공격자: /v1/alpha/freshness 를 200번 호출
결과: api_call_count=200 → FICO activity_score=200 (최대) → FICO 750+ → 에스크로 면제

[패치 후]
FICO 계산: api_calls.min(10_000) 적용, 저비용 엔드포인트는 0.5 가중치
결과: 비정상적 파밍으로 score 급등 불가
```

### 시나리오 D: DoS via 대형 payload (C7, C8)
```
[패치 전]
공격자: {"knowledge_context": ["A".repeat(1_000_000)] * 1000}
결과: 서버 OOM → 전체 다운

[패치 후]
RequestBodyLimitLayer(10MB) → 413 Payload Too Large
내부 검증: knowledge_context max 50 × 4096 chars
결과: 차단됨
```
